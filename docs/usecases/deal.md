# 案件ユースケース

## フェーズ定義

| フェーズ | 意味 |
|---|---|
| `proposal_prep` | 提案準備。案件化直後の初期状態 |
| `proposed` | 提案済み。顧客に提案書を提出した |
| `negotiation` | 交渉中。条件調整フェーズ |
| `won` | 受注。契約成立 |
| `lost` | 失注 |

## 遷移ルール

- 終端状態（`won`, `lost`）以外は自由に変更可能
- 終端状態からの遷移は不可
- 全フェーズから `lost` への遷移は可能（終端状態除く）
- フェーズ変更は admin / manager のみ

## 案件の作成パターン

### パターン1: 引き合い経由

- 引き合いが `in_progress` → 案件化 → `converted` に遷移 + Deal 自動作成
- 引き合いのタイトル・顧客を引き継ぎ

### パターン2: 直接作成

- 既存顧客からの追加依頼、口頭ベースの案件
- 引き合いを経ずに顧客を指定して直接 Deal を作成
- `clientId` 必須、`inquiryId` は null

## ユースケース詳細

### UC-01: 案件作成（引き合い経由）

- 前提: 引き合いが案件化される（inquiry UC-03）
- トリガー: 引き合いの案件化操作
- 処理: 引き合いのタイトル・顧客を引き継いで Deal を作成。初期フェーズは `proposal_prep`
- 結果: 案件が作成される

### UC-02: 案件作成（直接）

- 前提: 操作者が admin または manager
- 操作: 案件一覧から「新規作成」→ 顧客選択 + タイトル入力
- 処理: clientId 必須。inquiryId は null
- 結果: `proposal_prep` フェーズで案件が作成される

### UC-03: フェーズ変更

- 前提: 案件が終端状態でない。操作者が admin または manager
- 操作: フェーズ変更ボタン（次のフェーズ候補を表示）
- 結果: フェーズが変更される
- 備考: won になったら契約作成への導線を表示

### UC-04: 案件情報更新

- 前提: 案件が存在する。操作者が admin または manager
- 操作: タイトル・想定金額・期間・契約種別・担当者・備考の変更
- 結果: 案件情報が更新される

### UC-05: 失注

- 前提: 案件が終端状態でない。操作者が admin または manager
- 操作: 「失注」ボタン
- 結果: `lost` に遷移（終端）

### UC-06: 案件担当者管理

- 前提: 案件が存在する。顧客が確定している（clientId あり）
- 操作: 顧客の担当者を選択してロールを指定して追加/削除
