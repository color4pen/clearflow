# ユースケース

## このディレクトリの目的

ユーザーが達成したい目的（ユースケース）を記述する。「システムがどう動くか」（振る舞い）ではなく「ユーザーが何をしたいか」の単位で書く。

振る舞いの詳細（遷移ルール、バリデーション条件、副作用）はドメインサービスやユースケース層のコードで表現する。ここには書かない。

## 構造

- 1ドメイン1ファイル（`inquiry.md`, `deal.md`, `contract.md`）。ファイルが大きくなったら分割してよい
- ユースケースの粒度は「ユーザーゴール」レベルに揃える
  - 良い例: 「引き合いを案件化する」「契約を作成する」
  - 粗すぎ: 「受注を管理する」（サマリーゴール）
  - 細かすぎ: 「金額フィールドにカンマを表示する」（UI部品）
- 画面の部品配置や見た目の話は書かない。ユースケースはUIに依存しない
- 「どこから・どうやってたどり着くか」はナビゲーション（UI設計）の話であり、ユースケースではない
- 「一覧を表示する」「詳細を開く」単体はユースケースとして弱い。ユーザーの目的（判断・行動）があるかで判断する

## 運用

- ユースケースは育てるもの。完成させるものではない
- 実装やレビューで気づいた抜け漏れはすぐ追加する
- 変更コストを低く保つ。承認フローや厳密なステータス管理は設けない
- 役目を終えたユースケースは削除してよい。全量維持は目的ではない
- コードとの紐づけを意識する。ユースケース名とアプリケーション層の関数名を対応させると乖離に気づきやすい

## 書き方

各ユースケースには以下を含める:

- **目的**: ユーザーが何を達成したいか（1文）
- **前提**: この操作を行うための条件
- **権限**: 誰が実行できるか
- **結果**: 成功時に何が起きるか
- **制約**: 削除条件、上限チェック等のビジネスルール

振る舞いの詳細（ステータス遷移マップ、バリデーションロジック、エラーメッセージ）はここには書かない。それはコードの責務。

## 例

### 引き合いを案件化する

- **目的**: 見込みのある引き合いを案件として管理対象にする
- **前提**: 引き合いが存在すること
- **権限**: admin / manager
- **結果**: 案件が作成され、引き合いのステータスが変わる
- **制約**: 同一引き合いから複数案件は作成できない
