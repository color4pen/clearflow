# 認可設計

## 1. 概要

認可とは「認証済みのユーザーが、特定の操作を行う権限を持つか」を判定する仕組みである。本システムではロールベースアクセス制御（RBAC）を採用し、ユーザーのロールに基づいて操作の可否を決定する。

認可の設計原則:

- 認可ルールはドメイン層で一元的に定義する。アクション層やユースケース層に散在させない。
- すべての操作に対して認可が定義される。定義漏れは「拒否」として扱う。
- テナント境界（organizationId）の検証は認可とは別の関心事として、インフラストラクチャ層で保証する。

## 2. ロール定義

| ロール | 英語名 | 責務 |
|---|---|---|
| 管理者 | admin | システム全体の管理。ユーザー管理、設定変更、全操作の実行権限を持つ |
| マネージャー | manager | 営業チームの管理。案件の監督、承認、レポート確認を行う |
| 経理 | finance | 契約・請求・売上の管理。経理関連操作の実行権限を持つ |
| メンバー | member | 営業担当者。引合・案件・商談の日常操作を行う |

ロールは排他的であり、1 ユーザーは 1 つのロールのみを持つ。

## 3. 操作権限マトリクス

### 3.1 引合 (Inquiry)

| 操作 | admin | manager | finance | member |
|---|---|---|---|---|
| 一覧表示 | ○ | ○ | ○ | ○ |
| 詳細表示 | ○ | ○ | ○ | ○ |
| 作成 | ○ | ○ | − | ○ |
| 編集 | ○ | ○ | − | ○ |
| 案件化 | ○ | ○ | − | − |
| 見送り | ○ | ○ | − | − |
| 削除 | ○ | − | − | − |

- 案件化・見送りはマネージャー以上の判断が必要。
- メンバーは引合の登録・情報更新を行えるが、ステータス変更の判断は行えない。

### 3.2 案件 (Deal)

| 操作 | admin | manager | finance | member |
|---|---|---|---|---|
| 一覧表示 | ○ | ○ | ○ | ○ |
| 詳細表示 | ○ | ○ | ○ | ○ |
| 作成 | ○ | ○ | − | − |
| 編集 | ○ | ○ | − | ○ |
| フェーズ変更 | ○ | ○ | − | ○ |
| 受注・失注 | ○ | ○ | − | − |
| 担当者管理 | ○ | ○ | − | ○ |
| 削除 | ○ | − | − | − |

- 案件の作成はマネージャー以上（案件化の判断を含むため）。
- メンバーはフェーズ変更と情報更新を行えるが、終端フェーズ（受注・失注）への遷移は判断を伴うためマネージャー以上。

### 3.3 顧客接点 (Interaction)

UI/日本語表記は「商談」を維持する（kind = meeting の顧客接点）。権限値は変更しない。

| 操作 | admin | manager | finance | member |
|---|---|---|---|---|
| 一覧表示 | ○ | ○ | ○ | ○ |
| 詳細表示 | ○ | ○ | ○ | ○ |
| 作成 | ○ | ○ | − | ○ |
| 編集 | ○ | ○ | − | ○ |
| 削除 | ○ | ○ | − | − |

- 顧客接点（商談）は営業活動の記録であり、メンバーが日常的に作成・編集する。
- 上記の権限値は kind = meeting（商談）に対するもので、現行どおり変更しない。
- 顧客接点の「記録」は、契約・請求エンティティへの「操作」（作成・編集・ステータス変更。§3.5 / §3.6）とは別概念である。やり取りの事実を残す行為であり、操作権限とは別に kind ごとに定める。
  - kind = meeting（商談）: admin / manager / member。営業活動の記録。
  - kind = contract_adjustment（契約調整）: admin / manager / member。契約に関するやり取りは営業活動の延長であり、契約エンティティへの操作権限とは独立に定める。
  - kind = invoice_adjustment（請求調整）: admin / manager / finance。請求は経理領域であり、案件を統括する manager も記録しうる。

### 3.7 承認 (Approval)

| 操作 | admin | manager | finance | member |
|---|---|---|---|---|
| リクエスト一覧 | ○ | ○ | ○ | ○ |
| リクエスト詳細 | ○ | ○ | ○ | ○ |

メンバーは申請を行えるが、承認者としての役割は持たない。

## 4. 認可の判定フロー

```
1. リクエストを受信する
2. 認証情報からユーザーとロールを特定する
3. テナント境界を検証する（organizationId の一致）
4. 操作に対するロール権限を判定する
5. 操作固有の追加条件を検証する（※）
6. 許可 or 拒否
```

※ 追加条件の例:
- 承認操作: 該当ステップの承認者ロールとの一致、または委任の適用
- 案件の受注・失注: マネージャー以上のロール

## 5. テナント分離

認可とは別に、テナント分離はすべてのデータアクセスで保証されなければならない。

- すべてのクエリに `organizationId` のフィルタを適用する。
- ユーザーは自身が所属するテナントのデータにのみアクセスできる。
- テナント分離はインフラストラクチャ層（リポジトリ）で強制する。

## 6. 監査ログ

認可に関連する操作はすべて監査ログに記録する。

| 記録する情報 | 説明 |
|---|---|
| actorId | 操作を実行したユーザー |
| action | 実行した操作 |
| targetType | 操作対象の種別 |
| targetId | 操作対象の ID |
| organizationId | テナント |
| metadata | 操作固有の追加情報（変更前後の値など） |
| createdAt | 操作日時 |

監査ログの記録は、対象操作と同一トランザクション内で行う。
