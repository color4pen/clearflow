# 実装計画

設計と実装の乖離を解消するためのリクエスト計画。依存関係を考慮した実行順序で記載する。

## 依存関係グラフ

```
R01 ドメインイベント基盤
 ├→ R04 承認ポリシー＋システム連動承認（イベント基盤が必要）
 └→ R07 監査ログの自動化（イベント基盤が必要）

R02 ドメインモデル整備
 └→ R04 承認ポリシー（InquirySource に agent_service が必要）

R03 認可の一元化
 （独立して実行可能）

R04 承認ポリシー＋システム連動承認
 └→ R08 承認ポリシー設定画面（ポリシーが存在しないと画面が作れない）

R05 契約・請求モデル強化
 └→ R09 売上モジュール（契約金額が必須でないと売上集計が成立しない）

R06 ダッシュボード
 （独立して実行可能だが、R01 後のほうがイベントベースで効率的）

R07 監査ログの自動化
 （R01 に依存）

R08 承認ポリシー設定画面
 （R04 に依存）

R09 売上モジュール
 （R05 に依存）

R10 請求の独立画面化
 （独立して実行可能）
```

## リクエスト詳細

---

### R01: ドメインイベント基盤

**目的**: ドメイン層でイベントを定義・発行し、ハンドラで処理する仕組みを導入する。承認ポリシー評価、監査ログ自動記録、Webhook 配信の基盤となる。

**背景**: 現在ドメインイベントの概念が存在せず、監査ログは 40+ ユースケースに手動でコピペされ、Webhook はトランザクション外で fire-and-forget で呼ばれている。

**スコープ**:
- ドメインイベントの型定義（InquiryConverted, DealWon, DealLost, DealPhaseChanged, ContractCreated, ContractCompleted, ContractCancelled, InvoicePaid, InvoiceOverdue, InquiryDeclined）
- イベントディスパッチャー（同期ハンドラ / 非同期ハンドラの区別）
- 既存ユースケースへのイベント発行の組み込み（最低限 Inquiry, Deal, Contract, Invoice の主要遷移）
- Webhook 配信をイベントハンドラに移行

**スコープ外**:
- 監査ログのイベントハンドラ移行（R07 で実施）
- 承認ポリシー評価のハンドラ接続（R04 で実施）

---

### R02: ドメインモデル整備

**目的**: ドメイン設計書（01-domain-design.md）に合わせて、モデル・スキーマの不足と不整合を解消する。

**背景**: 型定義のみで振る舞いがないモデル、設計と乖離したスキーマ、不足している値オブジェクトが多数ある。

**スコープ**:
- Inquiry: budget, timeline カラム追加。InquirySource に email, agent_service を追加
- Meeting: inquiryId カラム追加（nullable）。CHECK 制約（dealId OR inquiryId が NOT NULL）。Attendee 構造を設計に合わせる（userId, contactId, name, isExternal）
- Deal: description カラム追加。DealPhase に estimate_approval を追加するか判断（設計書と要相談）
- Contract: amount を NOT NULL に変更（マイグレーションで既存 null データへのデフォルト値設定が必要）。startDate を NOT NULL に変更。CHECK 制約（startDate ≤ endDate）追加
- Invoice: issueDate カラム追加（invoicedAt との関係を整理）。CHECK 制約（issueDate ≤ dueDate）追加
- ClientContact: isPrimary のユニーク性をアプリケーション層で検証する関数追加
- 各モデルにファクトリ関数またはバリデーション関数を追加

**スコープ外**:
- Deal の extra フィールド（assigneeId, technicalLeadId 等）の削除判断は別途
- スキーマの version, updatedAt カラムは運用上有用なため残す

---

### R03: 認可の一元化

**目的**: アクション層に散在するロールチェックをドメイン層の認可ポリシーとして一元化し、設計書（03-authorization-design.md）の権限マトリクスに合わせる。

**背景**: finance ロールが全操作から除外されている、member が案件編集できない、削除権限が設計より緩い、など認可ルールが設計と大きく乖離している。

**スコープ**:
- ドメイン層に認可ポリシーモジュールを新設（ロール × エンティティ × 操作の権限定義）
- 全アクションファイルのロールチェックを認可モジュール呼び出しに置換
- 設計書の権限マトリクスに合わせた修正:
  - 契約操作に finance ロールを追加
  - 請求操作を admin + finance に修正（manager を除外）
  - 案件の編集・フェーズ変更に member を許可
  - 引合の編集に member を許可
  - 顧客担当者の追加・編集に member を許可
  - 削除操作を admin のみに制限
  - 委任操作を admin/manager/finance に開放
  - テンプレート一覧の閲覧を manager に許可
  - ユーザー一覧の閲覧を manager に許可

**スコープ外**:
- 承認操作の認可（承認ステップの承認者判定は既存の approvalStepService で処理済み）

---

### R04: 承認ポリシーとシステム連動承認

**目的**: 承認ポリシーエンティティを新設し、ドメインアクションに応じた自動承認リクエスト生成を実装する。

**背景**: 承認設計書（02-approval-design.md）の中核機能であるポリシーとシステム連動が丸ごと未実装。引合の案件化は承認なしで直接実行される。

**スコープ**:
- approval_policies テーブル新設（id, organizationId, name, description, triggerAction, conditionField, conditionOperator, conditionValue, templateId, isActive, createdAt）
- TriggerAction 列挙型（inquiry.convert, contract.create, contract.cancel）
- ComparisonOperator 列挙型（eq, neq, gt, gte, lt, lte, in）
- ApprovalRequest に origin フィールド追加（origin_type, origin_policy_id, origin_trigger_action, origin_trigger_entity_id）
- ApprovalStep に name, approverId カラム追加
- approval_delegations に from_user_role カラム追加
- ポリシー評価サービス（triggerAction に該当するポリシーを取得し、条件を評価する）
- ドメインイベントハンドラとしてポリシー評価を接続（InquiryConverted → ポリシー評価 → 承認リクエスト生成 or 即時実行）
- 承認完了時の承認後アクション実行（ApprovalCompleted イベント → triggerAction に応じた処理）
- updateInquiryStatus の案件化フローを承認ゲート対応に改修

**スコープ外**:
- 承認ポリシー設定画面（R08 で実施）
- contract.create, contract.cancel のポリシー連動（引合→案件化を先行実装し、パターンを確立してから横展開）

---

### R05: 契約・請求モデル強化

**目的**: 売上管理の基盤として、契約と請求のデータ整合性を強化する。

**背景**: 契約金額が nullable で売上集計の起点が崩れる。請求に issueDate がなく発行日の管理ができない。

**スコープ**:
- Contract: amount を NOT NULL に変更（既存データのマイグレーション含む）
- Contract: startDate を NOT NULL に変更
- Contract: CHECK 制約（startDate ≤ endDate、amount > 0）追加
- Invoice: issueDate カラム追加。既存の invoicedAt との整合（issueDate = 請求予定日、invoicedAt = 実際の発行日時として使い分けるか統合するか判断）
- Invoice: CHECK 制約（issueDate ≤ dueDate、amount > 0）追加
- 請求の更新時にも合計金額チェックを実施（現在は作成時のみ）

**スコープ外**:
- 売上集計ロジック（R09 で実施）
- 請求画面の独立化（R10 で実施）

---

### R06: ダッシュボード

**目的**: システムのエントリーポイントとして営業ダッシュボードと経理ダッシュボードを実装する。

**背景**: 現在 `/` が `/requests` にリダイレクトされており、ユーザーが今やるべきことを把握する画面がない。

**スコープ**:
- ダッシュボードページ（`/` or `/dashboard`）の新設
- ロールに応じたデフォルト表示の切替
- 営業ダッシュボード:
  - アクション待ちリスト（承認待ち + 未完了アクションアイテム + 未対応引合を統合、期日順）
  - パイプラインサマリ（フェーズ別の案件数と想定金額合計）
  - 直近の活動（監査ログベース）
  - manager/admin 向け: 担当者別分布、停滞案件リスト
- 経理ダッシュボード:
  - 期日超過の請求一覧
  - 未入金の請求一覧
  - 今月の売上サマリ
  - 請求予定
- グローバルナビゲーションにダッシュボードリンクを追加

**スコープ外**:
- 売上ダッシュボード（R09 で実施）
- 通知機能

---

### R07: 監査ログの自動化

**目的**: 監査ログの記録をドメインイベントハンドラに移行し、ユースケースからの手動記録を不要にする。

**背景**: 40+ ユースケースに同じ auditLogRepository.create パターンが繰り返されており、記録漏れのリスクがある。

**スコープ**:
- ドメインイベントの同期ハンドラとして監査ログ記録を実装
- 各イベントに対応する action, targetType, metadata のマッピング定義
- 既存ユースケースから手動の監査ログ記録コードを削除
- テストで監査ログが自動記録されることを検証

**スコープ外**:
- 監査ログの UI 変更
- 監査ログのスキーマ変更

---

### R08: 承認ポリシー設定画面

**目的**: 管理者が承認ポリシーを一覧・作成・編集・無効化できる設定画面を追加する。

**背景**: R04 でポリシーのバックエンドを実装するが、管理画面がないと設定できない。

**スコープ**:
- `/settings/policies` ページの新設
- ポリシー一覧テーブル（ポリシー名、トリガーアクション、条件、テンプレート名、有効/無効）
- ポリシー作成フォーム（ポリシー名、トリガーアクション、条件、テンプレート選択）
- ポリシー編集フォーム
- 有効/無効トグル
- SettingsNav にポリシーリンクを追加
- サーバーアクション: createPolicy, updatePolicy, listPolicies
- 認可: admin のみ作成・編集可能、admin/manager は一覧閲覧可能

**スコープ外**:
- ポリシーの削除（無効化で対応）
- 条件のプレビュー・テスト機能

---

### R09: 売上モジュール

**目的**: 売上ダッシュボード、売上明細、予実管理の 3 画面を実装する。

**背景**: 売上管理はシステムの最終目的の一つだが、画面もロジックも未実装。

**スコープ**:
- 売上ダッシュボードページ（`/revenue`）
  - 今月の入金確認済み合計
  - 月次推移グラフ（過去 12 ヶ月）
  - パイプラインベースの売上予測
  - 顧客別売上ランキング
- 売上明細ページ（`/revenue/details`）
  - 期間指定、集計軸切替（月別/顧客別/案件別）
  - CSV エクスポート
- 予実管理ページ（`/revenue/forecast`）
  - 期間ごとの目標金額設定
  - 進捗率表示
  - 着地予測
- 売上集計のユースケース・リポジトリ
- グローバルナビゲーションに売上リンクを追加

**スコープ外**:
- 目標金額の承認ワークフロー
- 売上予測のフェーズ別重み付け

---

### R10: 請求の独立画面化

**目的**: 請求を契約詳細への埋め込みから独立した画面に分離する。

**背景**: 画面仕様では請求は独立した詳細・登録画面を持つが、現在は契約詳細画面に埋め込まれている。

**スコープ**:
- 請求詳細ページの新設（契約詳細からリンクで遷移）
- 請求登録ページの新設（契約詳細から遷移、contractId 自動設定）
- 請求詳細画面のステータス操作（発行、入金確認、期日超過）
- 入金確認ダイアログ（入金日入力）
- 契約詳細の請求セクションは一覧表示 + リンクに簡素化

**スコープ外**:
- 請求一覧ページ（経理ダッシュボードで代替）
