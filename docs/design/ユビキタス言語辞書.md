## 監査アクション・対象種別の命名

監査ログの `AuditAction` と `AuditTargetType` は型カタログで一元管理する（正本コードは `src/domain/models/auditLog.ts`）。命名規約は次のとおり。

- **AuditAction**: `<対象>.<操作>` 形式。`<対象>` は `AuditTargetType` の識別子に準じ、複合語は snake_case（例: `deal`, `action_item`, `revenue_target`）。`<操作>` は camelCase（例: `create`, `updatePhase`, `updateStatus`, `conversionPending`）。
- **AuditTargetType**: エンティティ識別子。複合語は snake_case（例: `action_item`, `client_contact`, `deal_contact`, `revenue_target`）。

既知の逸脱（後方互換のため現状維持。新規のアクション・対象種別では踏襲しない）:

- `invoice.update_status` — 規約上は `invoice.updateStatus`（他は `contract.updateStatus` / `inquiry.updateStatus`）。
- AuditTargetType `approvalPolicy` — 規約上は `approval_policy`（他の複合語は snake_case）。

## 担当者の区別

| 区分 | 用語 | コード |
|---|---|---|
| 自社側 | 営業担当 | deal.assigneeId（ユーザー） |
| 自社側 | 技術担当 | deal.technicalLeadId（ユーザー） |

「担当者」とだけ書かず、自社側か先方側かを明示する。

## 表記の注意（混同しやすい用語）

- **商談 ≠ 会議**: 日本語表記は必ず「商談」。社内会議ではないので日本語で「会議」と書かない。`kind=meeting` はチャネル（会議）を指す語であり「商談」と同義ではない（定義は design/domain/glossary.md「商談」）。
- **案件 ≠ 商談**: 「案件（deal）」はパイプラインの単位、「商談」はその下の打ち合わせ記録。別物。
- **タスク = アクションアイテム**: UI 表記は「タスク」、ドメイン/コードは `action_item`。同一物。
