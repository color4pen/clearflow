# ユビキタス言語辞書

ドメイン用語の正本。日本語表記・コード識別子・意味を 1 対 1 で定める。UI・設計書・会話はこの日本語表記に統一し、コードはこのコード識別子に統一する。

## エンティティ

| 用語（日本語） | コード識別子（テーブル / 型） | 意味 |
|---|---|---|
| 引合 | inquiries / Inquiry | 顧客からの問い合わせ。案件化前のパイプラインの入口 |
| 案件 | deals / Deal | 受託案件。引合から案件化、または直接作成する。パイプラインの中核 |
| 顧客接点 | interactions / Interaction | 顧客との接点（商談・電話・メール・契約調整・請求調整…）の記録。`kind` で種別を表し、関連先は案件・引合・契約・請求・顧客のいずれか |
| 商談 | interactions（kind=meeting）/ Meeting | 顧客接点の一種。営業の打ち合わせ。商談種別はヒアリング / 提案 / 交渉 / クロージング / フォローアップ |
| 契約 | contracts / Contract | 案件の受注後に結ぶ契約 |
| 請求 | invoices / Invoice | 契約に対する請求 |
| 顧客 | clients / Client | 取引先企業 |
| 顧客担当者 | client_contacts / ClientContact | 顧客側の担当者（個人） |
| 案件担当者 | deal_contacts / DealContact | 案件に紐づく顧客側の担当者（役割付き） |
| アクションアイテム（タスク） | action_items / ActionItem | 案件・商談・引合に紐づくタスク。UI 上の呼称は「タスク」 |
| 承認リクエスト | requests / Request | 承認ワークフローの申請 |
| 承認ステップ | approval_steps / ApprovalStep | 承認リクエスト内の各承認段階 |
| 承認テンプレート | approval_templates / ApprovalTemplate | 承認フローのひな形 |
| 承認ポリシー | approval_policies / ApprovalPolicy | ドメインアクションに応じて承認を自動起動する条件定義 |
| 承認委任 | approval_delegations / ApprovalDelegation | 承認権限の一時委任 |
| 売上目標 | revenue_targets / RevenueTarget | 期間ごとの売上目標 |
| ウォッチ | watches / Watch | ユーザーが案件を購読し、配下の更新を通知で受け取る登録（UI ボタン表記は Watch / Watching） |
| 監査ログ | audit_logs / AuditLog | 全状態変更の追記専用記録 |

## 読み取りモデル（派生情報）

監査ログなどから読み取り時に構成する情報。永続化された独自テーブルを持たない。

| 用語（日本語） | コード識別子（型） | 意味 |
|---|---|---|
| 案件アクティビティ | DealActivityResult | 案件配下の顧客接点（商談）と業務イベントを監査ログから厳選・集約して時系列に構成したタイムライン。UI 呼称は「アクティビティ」（構成概念は後述「タイムラインの構成概念」） |
| 通知 | DerivedNotification | ウォッチ中の案件配下の更新を監査ログから導出した通知 |

## タイムラインの構成概念

案件などのタイムライン（UI 呼称「アクティビティ」）に何を表示するかを定める横断的な分類概念。独立したエンティティではなく、表示の選別基準として用いる。

| 概念 | 意味 | 例 | タイムライン |
|---|---|---|---|
| 顧客接点（Interaction） | 顧客との実際のやり取り。中身を持つ | 商談（meeting）・電話（call）・メール（email）・契約調整（contract_adjustment）・請求調整（invoice_adjustment） | 表示する |
| 業務イベント（Event） | 業務上意味のある節目・発生事実 | フェーズ変更・受注・失注・契約締結・請求発行・入金 | 表示する |
| 監査ログ（Log） | 全状態変更の網羅記録（追跡用） | あらゆる作成・更新・削除 | 無選別には流さない（厳選・集約の対象） |
| タスク（ActionItem） | やること（ToDo）。UI 呼称「タスク」 | アクションアイテム | 表示しない（別概念） |

- **タイムライン ＝ 顧客接点 ＋ 業務イベント の厳選表示**。監査ログを無選別に流さない。「アクティビティ」は UI 上の総称であり、独立したエンティティではない。
- 顧客接点は `interactions` テーブルで表す（商談は `kind=meeting`）。業務イベントは専用テーブルを持たず、監査ログから読み取って構成する。

## 監査アクション・対象種別の命名

監査ログの `AuditAction` と `AuditTargetType` は型カタログで一元管理する（正本コードは `src/domain/models/auditLog.ts`）。命名規約は次のとおり。

- **AuditAction**: `<対象>.<操作>` 形式。`<対象>` は `AuditTargetType` の識別子に準じ、複合語は snake_case（例: `deal`, `action_item`, `revenue_target`）。`<操作>` は camelCase（例: `create`, `updatePhase`, `updateStatus`, `conversionPending`）。
- **AuditTargetType**: エンティティ識別子。複合語は snake_case（例: `action_item`, `client_contact`, `deal_contact`, `revenue_target`）。

既知の逸脱（後方互換のため現状維持。新規のアクション・対象種別では踏襲しない）:

- `invoice.update_status` — 規約上は `invoice.updateStatus`（他は `contract.updateStatus` / `inquiry.updateStatus`）。
- AuditTargetType `approvalPolicy` — 規約上は `approval_policy`（他の複合語は snake_case）。

## 担当者の区別

| 区分 | 用語 | コード |
|---|---|---|
| 自社側 | 営業担当 | deal.assigneeId（ユーザー） |
| 自社側 | 技術担当 | deal.technicalLeadId（ユーザー） |
| 先方側 | 顧客担当者 / 案件担当者 | client_contacts / deal_contacts |

「担当者」とだけ書かず、自社側か先方側かを明示する。

## 表記の注意（混同しやすい用語）

- **商談 ≠ 会議**: 商談（kind = meeting の顧客接点）の日本語表記は必ず「商談」。種別が営業フェーズ（ヒアリング/提案/交渉/クロージング/フォローアップ）であり、社内会議ではない。日本語で「会議」と書かない。コード/テーブルは `interactions`（商談は kind = meeting）で表す。
- **案件 ≠ 商談**: 「案件（deal）」はパイプラインの単位、「商談（kind = meeting の interaction）」はその下の打ち合わせ記録。別物。
- **タスク = アクションアイテム**: UI 表記は「タスク」、ドメイン/コードは `action_item`。同一物。
