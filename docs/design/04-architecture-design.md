# アーキテクチャ設計

## 1. 概要

本システムは受託開発企業向けの案件管理 Web アプリケーションである。マルチテナント対応、承認ワークフロー、売上管理を中核機能とし、OSS として多様な受託企業が導入可能な構成を目指す。

## 2. 技術スタック

| 層 | 技術 | 選定理由 |
|---|---|---|
| ランタイム | Bun | 高速な起動・実行、TypeScript ネイティブ対応 |
| フレームワーク | Next.js (App Router) | SSR/RSC による高速な初期表示、Server Actions による型安全なサーバー通信 |
| 言語 | TypeScript (strict) | 型安全性によるドメインモデルの表現力 |
| ORM | Drizzle ORM | 型安全なクエリビルダー、軽量、マイグレーション管理 |
| データベース | PostgreSQL | ACID 準拠、JSON 型、列挙型、豊富な制約機能 |
| 認証 | Auth.js (next-auth) v5 | 柔軟なプロバイダ対応、セッション管理 |
| スタイリング | Tailwind CSS | ユーティリティファースト、デザインシステムとの親和性 |
| テスト | Bun test | ランタイム統合、高速な実行 |
| パッケージ管理 | Bun | ロックファイルの高速な解決 |

## 3. レイヤードアーキテクチャ

モジュール構成と層間の許可依存の正本は design/ にある。

- モジュール構成（責務・実装パス）: [design/static/modules.md](../../design/static/modules.md)
- 許可依存（列挙されない依存は禁止）: [design/static/dependencies.md](../../design/static/dependencies.md)

実装との一致は `src/__tests__/static/architecture.test.ts` が `design/rules.json` を用いて強制する。モジュールや依存を追加・変更するときは design/static/ を先に更新し、`bunx aozu export rules --out design/rules.json` で rules.json を再生成する。

以下は形式に載らない実装規約。

### 3.1 プレゼンテーション層の規約

- Server Components はデータ取得と表示を担う。読み取り専用のデータ取得に限り、リポジトリを直接参照してよい。状態を変更する操作は必ずアプリケーション層のユースケースを経由する。
- Server Actions は入力値の構造バリデーション（Zod）・認証状態の確認・レート制限を行い、ビジネスロジックは含めずユースケースに委譲する。

### 3.2 アプリケーション層の規約

- 1 ユースケースは 1 つの業務操作に対応し、1 トランザクションを原則とする。
- ユースケース間の直接呼び出しは行わない。共通のロジックはドメインサービスに抽出する。
- 戻り値は `Result` 型を返す。例外はインフラストラクチャ層の障害に限定する。
- 監査ログは `recordAudit` アプリケーションサービス経由で、操作の記録をトランザクション内に行う。

### 3.3 ドメイン層の規約

- 外部依存（DB・HTTP・ファイルシステム）を持たず、純粋な関数とデータ型で構成する。
- 入力値の構造バリデーションはプレゼンテーション層、業務的な検証はユースケースとドメインサービスで担う。

### 3.4 インフラストラクチャ層の規約

- リポジトリはすべての公開メソッドで organizationId によるテナント分離を強制する。

## 4. ドメインイベントの仕組み

ドメインイベントは、ドメインで発生した事象を他のコンテキストに伝達する仕組みである。

### 設計方針

- ドメインイベントは「業務上意味のある事実」（受注、請求の支払い、承認完了など）を表し、ユースケース層で発行する。
- イベントハンドラはアプリケーション/インフラ層で定義し、Webhook 配信・通知・承認ポリシー評価など、事実への波及反応を担う。
- 同期で処理すべきハンドラ（承認ポリシー評価）と、トランザクション後で処理してよいハンドラ（Webhook、通知）を区別する。

### ドメインイベントと監査ログの区別

監査ログ（全操作の記録）は**ドメインイベントのハンドラではない**。`recordAudit` アプリケーションサービス（`src/application/services/auditRecorder.ts`）を通じて、ユースケースのトランザクション内に直接記録する。記録する操作種別は型カタログ `AuditAction` / `AuditTargetType`（`src/domain/models/auditLog.ts`）で定義する。

- **ドメインイベント**: 業務上意味のある事実（`invoice.paid`, `deal.won` など）。Webhook・通知・承認などの波及反応を駆動する。
- **監査ログ**: 全操作の記録（`invoice.update_status`, `deal.update` など）。`recordAudit` で記録する。アクティビティ表示は、このうち業務的に意味のあるもの（顧客接点・業務イベント）を厳選・集約して読む（「ドメイン設計」§7.2 を参照）。

両者は別の語彙・別の関心事として扱い、混同しない。

### イベントの処理分類

| 分類 | 処理タイミング | 例 |
|---|---|---|
| 同期（トランザクション内） | ユースケースのトランザクション内 | 承認ポリシー評価 |
| 非同期（トランザクション後） | トランザクション完了後 | Webhook 配信、メール通知 |

### 承認ポリシーの評価

承認ポリシーの評価はドメインイベントの同期ハンドラとして実装する。

```
ユースケース: 引合を案件化する
  ├─ トランザクション開始
  ├─ ドメインイベント: InquiryConverted 発行
  ├─ 同期ハンドラ: 承認ポリシー評価
  │   ├─ 該当ポリシーなし → 案件を生成
  │   └─ 該当ポリシーあり → 承認リクエストを生成、案件は生成しない
  ├─ 監査ログ記録: recordAudit サービス（トランザクション内・イベントハンドラではない）
  ├─ トランザクション完了
  └─ 非同期ハンドラ: Webhook 配信、通知
```

## 5. トランザクション設計

### 原則

- 1 ユースケース = 1 トランザクション。
- トランザクション内で複数のリポジトリ操作と監査ログ記録を行う。
- 楽観的ロックまたはシリアライザブル分離レベルで競合を防ぐ。

### トランザクション境界の例

| ユースケース | トランザクション内の操作 |
|---|---|
| 引合の案件化 | 引合ステータス更新 + ポリシー評価 + (案件生成 or 承認リクエスト生成) + 監査ログ |
| 承認の実行 | ステップ更新 + リクエストステータス更新 + (承認後アクション実行) + 監査ログ |
| 請求の作成 | 請求作成 + 契約金額の整合性チェック + 監査ログ |

## 6. エラーハンドリング

### 分類

| 種別 | 例 | 処理方針 |
|---|---|---|
| バリデーションエラー | 入力値の形式不正、必須フィールド未入力 | Server Action 層で検出し、ユーザーにフィードバック |
| ビジネスルール違反 | 不正な状態遷移、金額超過 | ユースケースから Result 型で返却 |
| 認可エラー | 権限不足 | ユースケースで判定し、Result 型で返却 |
| インフラ障害 | DB 接続エラー、タイムアウト | 例外として伝播し、プレゼンテーション層でエラー画面を表示 |
| 競合エラー | 楽観的ロックの衝突 | リトライまたはユーザーへの通知 |

### Result 型

ユースケースの戻り値は成功と失敗を型で区別する。

```typescript
type Result<T> =
  | { ok: true; data: T }
  | { ok: false; reason: string }
```

## 7. マルチテナント

### 分離戦略

共有データベース・共有スキーマ方式を採用する。すべてのテーブルに `organizationId` カラムを持ち、クエリレベルでテナント分離を行う。

| 層 | テナント分離の実装 |
|---|---|
| リポジトリ | すべてのクエリに `organizationId` フィルタを自動適用 |
| ユースケース | セッションから取得した `organizationId` をリポジトリに渡す |
| Server Action | セッションから `organizationId` を取得し、ユースケースに渡す |

### テナント分離の保証

- リポジトリのすべての公開メソッドは `organizationId` を必須パラメータとする。
- テナントをまたぐデータアクセスは API レベルで不可能とする。

## 8. バッチ処理

定期的に実行が必要な処理。

| 処理 | 頻度 | 内容 |
|---|---|---|
| 承認期限切れ検知 | 日次 | 期限を超過した承認ステップを検出し、リクエストを expired に遷移 |
| 請求期日超過検知 | 日次 | 支払期日を超過した請求を検出し、overdue に遷移 |

バッチ処理は Cron エンドポイント（認証付き API）として実装し、外部のスケジューラから呼び出す。

## 9. 監査ログ

すべての状態変更操作を記録する。

### 設計方針

- 監査ログの記録は、対象操作と同一トランザクション内で行う。
- 監査ログは追記のみ。更新・削除は行わない。
- 記録は `recordAudit` アプリケーションサービスを通じてユースケースのトランザクション内に直接行う（ドメインイベントのハンドラ経由ではない。「ドメインイベントと監査ログの区別」を参照）。

### 記録対象

| 対象 | アクション例 |
|---|---|
| 引合 | created, updated, converted, declined |
| 案件 | created, updated, phase_changed, won, lost |
| 契約 | created, updated, completed, cancelled |
| 請求 | created, updated, invoiced, paid, overdue |
| 承認 | requested, step_approved, step_rejected, approved, rejected, expired |
| ユーザー | role_changed |
| 設定 | template_created, policy_created, webhook_created |

## 10. Webhook

外部システムへのイベント通知。

### 設計方針

- Webhook の配信はドメインイベントの非同期ハンドラとして実装する。
- 配信はトランザクション完了後に行い、DB の状態変更との整合性を保つ。
- 配信失敗時はリトライする（指数バックオフ、最大回数制限）。
- 配信結果（成功/失敗、レスポンスコード、レスポンスボディ）を記録する。

### Webhook エンドポイント

テナントごとに Webhook エンドポイントを設定できる。

| 属性 | 説明 |
|---|---|
| url | 配信先 URL |
| events | 購読するイベントの種別 |
| secret | 署名検証用のシークレット |
| isActive | 有効/無効 |

### 配信イベント

ドメインイベントと同じイベント種別を Webhook で配信する。ペイロードにはイベント種別、対象エンティティの ID、タイムスタンプを含む。
