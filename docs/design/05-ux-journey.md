# ユーザー体験設計

## 1. 概要

本書はこのシステムを使う人の体験を設計する。「どの画面に何の機能があるか」ではなく、「どういう場面で何を考え、システムがどう応えるか」を定義する。

## 2. 体験設計の原則

### 2.1 今やるべきことが見える

ユーザーがシステムを開いたとき、最初に目に入るのは「自分が今やるべきこと」でなければならない。データの一覧ではなく、行動の起点を提供する。

### 2.2 文脈は引き継ぐ

ユーザーの頭の中にある文脈を、画面遷移で途切れさせない。引合から案件を作る、案件から契約を作る——前の画面で見ていた情報は次の画面にも引き継がれる。入力のやり直しは文脈の断絶であり、避ける。

### 2.3 判断に必要な材料を手元に置く

判断を求める場面では、その判断に必要な情報が同じ画面に揃っていなければならない。「別の画面を見に行かないと判断できない」状態はシステムの欠陥である。

### 2.4 不可逆な操作は慎重に、日常操作は軽快に

受注・失注の確定、契約の解除のような不可逆な操作は確認を求める。一方、商談の記録やフェーズの変更のような日常操作は、最小限の手順で完了できるようにする。

### 2.5 ロールによって見え方が変わる

営業担当は自分の案件に集中したい。マネージャーはチーム全体を見たい。経理は契約と請求だけが関心事。同じシステムでもロールによって最適な情報の見せ方は異なる。ナビゲーションは共通でも、ダッシュボードや一覧のデフォルト表示はロールに応じて変える。

## 3. ユーザージャーニー

### Journey 1: 引合を受けて案件化するまで

**登場人物**: 営業担当（member）、マネージャー（manager）

**場面**: 仲介サービス経由で開発案件の引合が入った。ヒアリングを経て、案件として進めるか判断する。

---

**Step 1. 引合を登録する（営業担当）**

電話やメールで引合を受けた営業担当が、まず情報を記録する。

- この時点では顧客情報が不明確なこともある。件名と経路さえ入力すれば登録でき、詳細は後から追記できる。
- 顧客がわかっている場合は既存顧客を選択する。初めての顧客であれば、引合の登録の流れの中で新しい顧客を作成できる。別の画面に移動する必要はない。

**Step 2. ヒアリングを記録する（営業担当）**

顧客と初回の打合せを行い、課題・予算・時期・意思決定者・競合の情報を聞き出す。

- 引合詳細画面から「商談を追加」すると、紐づけ先が自動設定された状態で記録画面が開く。
- 商談種別を「ヒアリング」にすると、BANT 情報の入力欄が現れる。これは構造化された情報であり、後から案件化の判断材料として一覧できる。
- 出席者を記録する。初めて登場する顧客側の参加者は、その場で名前を入力して後から顧客担当者として正式に登録できる。
- 商談から生まれた TODO（アクションアイテム）をその場で追加し、担当者と期日を設定する。

**Step 3. 案件化を判断する（マネージャー）**

ヒアリング結果を踏まえて、案件として進めるか判断する。

- 引合詳細画面には、これまでの商談記録とそこで得られた情報が時系列で並んでいる。マネージャーはここを見て判断できる。別の画面に移動する必要はない。
- 「案件化する」を選択する。このとき:
  - 顧客が未設定の場合はボタンが無効になっており、理由が表示されている。
  - 仲介サービス経由の引合の場合、承認ポリシーによって自動的に承認リクエストが作られることがある。この場合は「承認リクエストを作成しました」と表示され、承認が通るまで案件は生成されない。
  - 承認が不要な場合は、即座に案件が生成され、案件詳細画面に遷移する。

**Step 4. 承認を処理する（別のマネージャーや管理者）**

承認が必要な場合、承認者のダッシュボードに承認待ちの項目として表示される。

- 承認者は承認リクエストの詳細を開くと、引合の情報（経路、顧客、ヒアリング内容）へのリンクが表示されている。判断材料に直接アクセスできる。
- 承認すると、案件が自動生成され、申請者に通知が届く。

---

### Journey 2: 案件を進めて受注する

**登場人物**: 営業担当（member）、マネージャー（manager）

**場面**: 案件化された後、提案・交渉を経て受注に至る。

---

**Step 1. 案件の状況を確認する（営業担当）**

営業担当はダッシュボードを開く。

- 「アクション待ち」に、期日の近いアクションアイテム（前回の商談で決まった TODO）が表示されている。ここが今日の仕事の起点になる。
- アクションアイテムをクリックすると、それが生まれた商談→所属する案件へと辿れる。

**Step 2. 提案・交渉を進める（営業担当）**

案件詳細画面が営業活動のホーム画面になる。

- フェーズは自由に変更できる。提案書を作ったら「提案準備→提案」、見積の話になったら「交渉」——実態に合わせて柔軟に動かせる。フェーズ変更は日常操作であり、確認ダイアログは出ない。
- 商談のたびに記録を追加していく。案件詳細画面には商談の時系列と、全商談から集約されたアクションアイテムが一覧で表示される。「次に何をすべきか」が常に見える。
- 案件配下で起きたこと——商談などの顧客接点と、フェーズ変更・受注・契約作成・請求といった業務上の節目——は、アクティビティ（タイムライン）として時系列で確認できる。日々の細かな更新やタスクの増減は流さず、意味のある動きだけを追える。
- 案件に関わる顧客側の関係者が増えたら、DealContact として追加し、役割（キーパーソン、意思決定者、技術担当）を設定する。提案書の送付先や、交渉の際に誰を押さえるべきかの判断に使える。
- 注視したい案件は「ウォッチ」しておくと、その案件配下の更新（フェーズ変更・商談やアクションアイテムの追加など）を通知で追える。自分が直接触らない案件でも状況を見逃さない。

**Step 3. 受注を確定する（マネージャー）**

交渉がまとまり、受注が確定する。

- マネージャーが案件詳細画面で「受注」を選択する。受注は不可逆な操作なので確認ダイアログが表示される。
- 受注が確定すると、案件詳細画面に「契約を作成する」という導線が現れる。次にやるべきことが自然に示される。

---

### Journey 3: 契約から請求、入金確認まで

**登場人物**: マネージャー（manager）、経理（finance）

**場面**: 受注した案件に対して契約を締結し、請求を行い、入金を確認する。

---

**Step 1. 契約を作成する（マネージャー）**

案件詳細画面の「契約を作成する」から遷移する。

- 案件が自動で紐づけられ、契約形態は案件の想定契約形態がデフォルト値として入っている。入力の手間を省く。
- 金額は必須。契約金額がこの先の請求と売上管理の起点になる。
- 1 つの案件に複数の契約を作れる。開発フェーズごと、保守契約、追加スコープなど、実態に合わせて分割できる。
- 承認ポリシーにより承認が必要な場合はその旨が表示される。

**Step 2. 請求を作成する（経理）**

経理は契約詳細画面から請求を作成する。

- 契約が自動で紐づく。単発契約の場合は「契約金額のうち残り○○円が請求可能」と表示される。超過する金額は入力できない。
- 請求日と支払期日を設定する。月末締め翌月末払いなどの慣行に合わせる。

**Step 3. 請求を発行する（経理）**

請求日が来たら、経理が「発行済」に変更する。

- 経理ダッシュボードに「本日が請求日の請求」が表示されている。見落としを防ぐ。

**Step 4. 入金を確認する（経理）**

入金があったら「入金済」に変更し、入金日を記録する。

- 経理ダッシュボードの「未入金の請求」に支払期日順で表示されている。
- 支払期日を過ぎた請求は自動で「期日超過」になり、目立つように表示される。
- 入金確認は売上実績に直結する操作であり、売上ダッシュボードにリアルタイムで反映される。

---

### Journey 4: マネージャーがパイプラインを管理する

**登場人物**: マネージャー（manager）

**場面**: 週次ミーティングに向けて、チームの営業状況を確認する。

---

**Step 1. パイプラインの全体像を見る**

案件一覧をフェーズ別のビューで表示する。

- フェーズごとの案件数と想定金額の合計が表示される。「交渉中の案件が全部決まればいくらの売上になるか」が一目でわかる。
- 受注見込み時期が近い案件はハイライトされている。今月クローズ予定の案件に注目できる。

**Step 2. 停滞案件を発見する**

ダッシュボードに「長期間フェーズが変わっていない案件」が表示されている。

- どの案件がどれだけ停滞しているかがわかる。案件をクリックすると詳細に飛び、最後の商談記録と残っているアクションアイテムを確認できる。
- 営業担当に声をかけるか、自分で介入するかの判断材料が揃っている。

**Step 3. 承認を処理する**

ダッシュボードの「承認待ち」に自分が承認すべきリクエストが表示されている。

- 各リクエストに対して、判断に必要な情報（引合の詳細、契約の金額など）へのリンクがある。
- その場で承認・却下ができる。コメントを添えることもできる。

---

### Journey 5: 経理が月次の売上を確認する

**登場人物**: 経理（finance）

**場面**: 月末に売上実績を集計し、経営層に報告する。

---

**Step 1. 今月の売上実績を確認する**

売上ダッシュボードを開く。

- 今月の入金確認済み合計が表示されている。
- 月次推移のグラフで、前月・前年同月との比較ができる。
- 顧客別売上ランキングで、主要顧客への依存度がわかる。

**Step 2. 来月以降の見込みを把握する**

- パイプライン上の案件（フェーズ × 想定金額）から、将来の売上見込みが算出されている。
- 確定済み（契約あり・請求予定あり）と見込み（案件フェーズベース）が区別されて表示される。
- 予実管理画面で、目標に対する進捗率を確認できる。

**Step 3. 明細をエクスポートする**

- 売上明細画面で、月別・顧客別・案件別に絞り込んで CSV エクスポートできる。
- 経理ソフトや社内報告フォーマットに取り込める。

---

### Journey 6: 管理者が承認ルールを設定する

**登場人物**: 管理者（admin）

**場面**: 新しく取引を始める仲介サービス経由の引合について、案件化時に経営層の承認を必要とするルールを追加する。

---

**Step 1. 承認テンプレートを作成する**

設定画面でテンプレートを作る。

- 「仲介案件承認」というテンプレートを作成し、ステップを定義する:
  1. マネージャー承認（期限: 3 日）
  2. 管理者承認（期限: 5 日）
- ステップの順序、承認者ロール、期限を設定するだけ。テンプレート自体は「どう承認するか」しか定義しない。

**Step 2. 承認ポリシーを作成する**

テンプレートとは別に、「いつ承認が必要か」を定義する。

- 「仲介案件の承認」というポリシーを作成する:
  - トリガー: 引合の案件化
  - 条件: 引合経路が「仲介サービス」
  - テンプレート: 仲介案件承認
- ポリシーを有効化すると、以降の仲介サービス経由の引合は案件化時に自動で承認リクエストが作られる。

**Step 3. 設定を確認する**

ポリシー一覧で、現在有効なすべてのポリシーを確認できる。

- どのアクションに対して、どの条件で、どのテンプレートが適用されるかが一覧で見える。
- 「承認がどこで発生するか」が暗黙知ではなく、明示的に管理されている。

---

### Journey 7: 承認者が不在のとき

**登場人物**: マネージャー A（委任元）、マネージャー B（委任先）

**場面**: マネージャー A が休暇を取る。その間の承認が滞らないよう、マネージャー B に権限を委任する。

---

**Step 1. 委任を設定する（マネージャー A）**

設定の承認委任画面で、委任を作成する。

- 委任先にマネージャー B を選択し、休暇期間を設定する。
- 委任が有効な間、マネージャー A 宛の承認ステップをマネージャー B が処理できるようになる。

**Step 2. 委任された承認を処理する（マネージャー B）**

マネージャー B のダッシュボードに、委任された承認リクエストが表示される。

- 通常の承認操作と同じように処理できる。
- 誰の委任で承認しているかが記録に残る。

## 4. 画面構成

ジャーニーを実現するための画面構成。

### ナビゲーション

```
ダッシュボード（ロール別）
├── 引合
├── 案件
├── 顧客
├── 契約
├── 承認
├── 売上
└── 設定
```

### ダッシュボード

ロールごとに異なるデフォルト表示を提供する。

| ロール | 主な表示内容 |
|---|---|
| 営業担当 | アクション待ち、自分の案件パイプライン、直近の活動 |
| マネージャー | 承認待ち、チーム全体のパイプライン、停滞案件、担当者別分布 |
| 経理 | 未入金の請求、期日超過、今月の売上サマリ、請求予定 |
| 管理者 | 承認待ち、直近の監査ログ、ユーザー管理への導線 |

### 一覧画面の共通パターン

- フィルタとソートで目的のデータに素早く到達できる。
- 一覧のデフォルト表示はロールと文脈に応じて最適化する（営業担当は自分の案件を先頭に、経理は期日順の請求を先頭に等）。
- 一覧から直接、重要な情報（ステータス、金額、期日）が読み取れる。詳細画面を開かなくても概況がわかる。

### 詳細画面の共通パターン

- 基本情報と関連エンティティを 1 画面で確認できる。
- 関連エンティティへのリンクを提供し、情報のトレーサビリティを確保する。
- ステータス変更や関連エンティティの追加は詳細画面内で完結する。別画面への遷移を最小化する。

### 登録・編集の共通パターン

- 必須項目を最小限にし、登録のハードルを下げる。詳細は後から追記できる。
- 文脈から自動設定できる項目（紐づけ先、デフォルト値）は自動設定する。
- 関連エンティティの新規作成は、画面遷移なしでインラインで行える（新規顧客の登録、外部参加者の追加等）。

## 5. 情報のトレーサビリティ

このシステムのデータは引合→案件→契約→請求→売上と流れていく。ユーザーがこの流れのどこにいても、上流と下流の情報を辿れなければならない。

| 起点 | 辿れる情報 |
|---|---|
| 引合詳細 | 商談記録、案件（案件化済みの場合）、顧客 |
| 案件詳細 | 元の引合、商談記録（引合フェーズのものも含む）、顧客、顧客側関係者、契約（受注後）、アクションアイテム |
| 契約詳細 | 案件、顧客、請求一覧 |
| 請求詳細 | 契約、案件、顧客 |
| 売上明細 | 請求、契約、案件、顧客 |
| 顧客詳細 | 引合一覧、案件一覧、売上実績 |

どのエンティティからでも、関連するエンティティの全体像に到達できる。

## 6. 通知と気づき

ユーザーが能動的に画面を見に行かなくても、重要な変化に気づける仕組み。

| 通知 | 対象者 | タイミング |
|---|---|---|
| 承認リクエストが届いた | 承認者 | リクエスト作成時 |
| 承認が完了した / 却下された | 申請者 | ステータス変更時 |
| 承認が期限切れになった | 申請者・承認者 | 期限到達時 |
| 請求の支払期日が近い | 経理 | 期日の N 日前 |
| 請求が期日超過になった | 経理 | 期日到達時 |
| 案件が長期間停滞している | 担当者・マネージャー | 一定期間フェーズ変更なし |
| アクションアイテムの期日が近い | 担当者 | 期日の N 日前 |

通知の手段（画面内通知、メール、Webhook）は段階的に拡張する。上表は対象とする気づきの全体像であり、一部は将来の拡張を含む。

**実装済みの仕組み: 案件ウォッチによる通知**

ユーザーが関心のある案件を「ウォッチ」しておくと、その案件配下の更新（フェーズ変更・案件情報の更新・商談の追加・アクションアイテムの追加・契約の追加）を、サイドバー下部の通知から受け取れる。

- 通知は監査ログから読み取って構成する（専用の通知レコードは持たない）。
- 自分自身の操作は通知に含めない。
- 未読はユーザーごとの最終確認時刻で判定し、「既読にする」で既読化する。

メール・Webhook 等の他チャネルや、承認・請求・停滞などウォッチ以外を起点とする通知は将来の拡張とする。
