# Design: invoice-management

## Context

契約（Contract）が作成・管理されているが、契約に紐づく請求情報を管理する仕組みがない。分割請求（着手金・中間金・残金など）を管理し、請求のライフサイクル（予定→請求済→入金済/期日超過）を追跡する必要がある。

既存の構造:
- `contracts` テーブルに `amount`（nullable integer）と `renewalType`（`one_time | recurring`）が存在する
- 全テーブルに `organizationId` によるテナント分離が適用されている
- リポジトリ→ユースケース→Server Actions のレイヤードアーキテクチャが確立されている
- 契約詳細ページ（`/contracts/[id]`）が存在し、契約情報と関連情報を表示している
- 監査ログ（`auditLogRepository.create`）による操作記録パターンが確立されている

## Goals / Non-Goals

**Goals**:

- 契約に紐づく請求（Invoice）の CRUD を実現する
- 請求ステータスの遷移管理（scheduled → invoiced → paid / overdue）を実装する
- one_time 契約で請求合計が契約金額を超えないバリデーションを実装する
- 契約詳細ページ内に請求一覧セクション・サマリー・追加フォーム・ステータス変更UIを配置する
- 全操作に対して監査ログを記録する

**Non-Goals**:

- 請求書の PDF 生成・出力
- 入金管理の詳細（部分入金、振込手数料）
- 定期契約の自動請求生成
- 請求と承認ワークフローの連携
- 請求の金額編集（作成後は金額固定）
- 独立した請求一覧ページ（契約詳細ページ内で完結）

## Decisions

### D1: 請求ステータス遷移を有限状態マシンで制御

**選択**: `scheduled → invoiced → paid | overdue` の固定遷移ルール。`paid` と `overdue` は終端状態。`scheduled` からは `invoiced` のみ許可。

**理由**: ステータス遷移を明示的にルール化することで、不正な状態変更を防止する。既存の `requestTransition.ts` の `VALID_TRANSITIONS` パターンと一貫性がある。

**却下した代替案**: 自由な遷移を許可してバリデーションで制御 — 遷移ルールが散在し、保守性が低下する。

### D2: ステータス遷移時に日時フィールドを自動セット

**選択**: `scheduled → invoiced` 遷移時に `invoicedAt` を現在日時にセット。`invoiced → paid` 遷移時に `paidAt` を現在日時にセット。ユースケース内で自動的に設定し、呼び出し側に委ねない。

**理由**: 日時の一貫性を保証する。手動入力にすると入力ミスや偽装のリスクがある。

**却下した代替案**: UI から日時を入力させる — 監査上の信頼性が低下する。

### D3: one_time 契約のみ合計金額チェックを実施

**選択**: `createInvoice` で `contract.renewalType === "one_time"` の場合のみ、`sumAmountByContract` で既存請求合計を取得し、新規請求金額との合計が `contract.amount` を超えないことを検証する。`contract.amount` が null の場合はチェックをスキップする。recurring 契約では合計金額チェックをスキップする。

**理由**: recurring 契約は期間に応じて請求が無限に追加されるため、合計金額の上限を設けると運用が破綻する（architect 評価済み）。

**代替案なし**: 要件で明示されている。

### D4: sumAmountByContract を DB レベルの SUM で実装

**選択**: `invoiceRepository.sumAmountByContract(contractId, organizationId)` として、SQL の `SUM(amount)` を使用する。

**理由**: 全請求を JS に取得して集計する方式はデータ量に比例してメモリ・ネットワーク負荷が増大する。DB レベルの集計はパフォーマンスが良い（architect 評価済み）。

**却下した代替案**: 全請求取得後に JS で `reduce` — 大量の請求がある場合に非効率。

### D5: 請求管理を契約詳細ページ内に配置

**選択**: 独立した `/invoices` ページを作らず、`/contracts/[id]` ページ内に請求一覧セクション・サマリー・追加フォーム・ステータス変更UIを配置する。

**理由**: 請求は常に契約のコンテキストで閲覧・操作する。独立ページだと契約との関係が見えにくい（architect 評価済み）。

**却下した代替案**: 独立した請求一覧ページ — 契約との関係が分断される。

### D6: 請求追加フォームをモーダルで実装

**選択**: 契約詳細ページ内の「請求を追加」ボタンからモーダルダイアログを表示し、タイトル・金額・支払期日・備考を入力する。

**理由**: ページ遷移なしで請求を追加でき、契約詳細のコンテキストを維持できる。インラインフォームよりもレイアウトの制約が少ない。

**却下した代替案**: 別ページへの遷移 — 契約コンテキストが失われる。インラインフォーム — レイアウトが煩雑になる。

## Risks / Trade-offs

- **[Risk] one_time 契約で contract.amount が null のケース** → 合計金額チェックをスキップする。契約金額未設定の場合は上限なしとして動作する。
- **[Risk] 契約ステータスが active 以外の場合の請求作成** → ユースケースで契約が `active` であることを検証し、非 active 契約への請求作成を拒否する。
- **[Trade-off] overdue の手動遷移** → 支払期日超過の自動検出は cron ジョブが必要であり、初期段階では過剰。手動遷移で開始し、将来的に自動化を検討する。

## Open Questions

なし（architect 評価済みの設計判断で全主要論点が解決済み）。
