## 背景

D01 でテーマ基盤とサイドバーレイアウトを適用した。本リクエストでは営業ダッシュボードと経理ダッシュボードの画面レイアウトを Claude Design（docs/design/Clearflow.dc.html の SALES DASHBOARD / FINANCE DASHBOARD セクション）に合わせる。

## 現状コードの前提

- `src/app/(dashboard)/dashboard/page.tsx` — ダッシュボードのエントリーポイント。ロール判定で SalesDashboard / FinanceDashboard を切り替え
- `src/app/(dashboard)/dashboard/SalesDashboard.tsx` — 営業ダッシュボード。PageToolbar + HTML テーブルベースの表示。パイプラインサマリなし。アクション待ちリストは 4 カラムテーブル。直近の活動は 3 カラムテーブル。停滞案件は 3 カラムテーブル
- `src/app/(dashboard)/dashboard/FinanceDashboard.tsx` — 経理ダッシュボード。PageToolbar + 縦積みの SectionCard。KPI カードグリッドなし。請求テーブルは 4 カラム（Title, Amount, DueDate, ContractLink）
- `docs/design/Clearflow.dc.html:66-94` — デザインの営業ダッシュボード: ヘッダー（タイトル + 日付/ロール + アクションボタン2つ）+ 6カラムパイプライングリッド + 2カラムメインコンテンツ（左: アクション待ちリスト、右: 停滞案件 + 直近の活動）
- `docs/design/Clearflow.dc.html:152-225` — デザインの経理ダッシュボード: ヘッダー + 4カラム KPI グリッド + 期日超過テーブル（6カラムグリッド）+ 2カラム（未入金 + 請求予定）

## 要件

1. **営業ダッシュボード — ヘッダー**: PageToolbar を廃止し、ダッシュボード専用ヘッダーに変更する。左: ページタイトル「ダッシュボード」+ サブテキスト（日付 + ロール）。右: 「案件を見る」（outline ボタン、/deals へ遷移）+ 「引合を登録」（primary ボタン、/inquiries/new へ遷移）
2. **営業ダッシュボード — パイプラインサマリ**: 6 カラムのグリッド（5 フェーズ + 合計）。各セルにフェーズ名、件数（大きめフォント + 「件」）、金額（mono フォント）。セルをクリックすると /deals?phase=xxx に遷移。SectionCard でラップし、ボーダーで区切る
3. **営業ダッシュボード — メインコンテンツ 2 カラム**: 左カラム（1.55fr）にアクション待ちリスト、右カラム（1fr）に停滞案件 + 直近の活動。gap は 24px
4. **営業ダッシュボード — アクション待ちリスト**: テーブルを廃止し flex レイアウトに変更する。各行: 左にタイプラベル（承認待ち / アクション / 新規引合。62px 固定幅、色分け）、中央に件名 + サブテキスト（案件名等）、右に日付（超過は赤文字 + 「超過」ラベル）。ヘッダーに超過件数バッジ（赤）と件数表示
5. **営業ダッシュボード — 停滞案件**: 案件名 + 停滞日数（赤文字）。サブテキストにフェーズ・金額・担当者をドット区切りで表示
6. **営業ダッシュボード — 直近の活動**: flex レイアウト。担当者名（46px 幅）+ 操作テキスト + 相対時間（text-muted）
7. **経理ダッシュボード — ヘッダー**: 営業と同じ構造。ボタンは「契約を見る」（outline、/contracts へ遷移）
8. **経理ダッシュボード — KPI カードグリッド**: 4 カラムグリッド（1.4fr 1fr 1fr 1fr）。1: 今月の売上（金額を大きめフォント、green #1a8a4a）。2: 期日超過（件数、赤 #c0392b）。3: 未入金（件数）。4: 請求予定（件数）
9. **経理ダッシュボード — 期日超過テーブル**: 6 カラムグリッド（請求番号, 契約名, 顧客名, 金額, 支払期日, 超過日数）。金額と日付は mono フォント。超過日数は赤文字
10. **経理ダッシュボード — 下部 2 カラム**: 左: 未入金の請求（flex レイアウト、契約名 + 顧客名 + 期日 + 金額）。右: 請求予定（同レイアウト）

## architect 評価済みの設計判断

1. **テーブルを flex/grid レイアウトに置換** — デザイン HTML がテーブルではなく flex/grid を使用しているため、それに合わせる。HTML テーブルは固定幅の制御が難しく、デザインの細かいレイアウト指定に合わない。却下案: テーブルのスタイル修正のみ — デザインの列幅比率（1.55fr:1fr 等）を再現できない
