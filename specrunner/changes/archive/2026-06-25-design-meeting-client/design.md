# Design: 商談・顧客画面のデザイン適用

## Context

Claude Design の MEETING DETAIL / CLIENT DETAIL / CLIENTS LIST セクションに合わせ、3 画面のレイアウトを更新する。現状とデザインの主な乖離:

- **商談詳細**: カラム比率が `1fr 2fr`（左:基本情報、右:議事録+アクションアイテム）だが、デザインは `1.6fr 1fr`（左:議事録+ヒアリング、右:出席者+アクションアイテム）。左右のセクション配置が逆転する
- **商談詳細 MeetingInfoSection**: 基本情報・出席者・ヒアリング情報が 1 コンポーネントに統合されているが、デザインでは左右カラムに分散するため分割が必要
- **顧客一覧**: 7 カラム表示だが、デザインは 4 カラム（企業名, 業種, 関連案件数, 登録日）に絞っている
- **顧客詳細**: `grid-cols-2`（等分）で担当者・関連エンティティがフルワイズ配置だが、デザインは `1.5fr 1fr` で 2 カラム内に収める構成

変更はレイアウトと表示のみ。ビジネスロジック・登録フォームは変更しない。

## Goals / Non-Goals

**Goals**:

- 商談詳細のカラム比率を `1.6fr 1fr`、gap 24px に変更し、議事録を左・出席者を右に配置する
- MeetingInfoSection を 3 コンポーネント（基本情報、出席者、ヒアリング）に分割する
- 顧客一覧を 4 カラムに削減し、不要なデータ取得を除去する
- 顧客詳細を `1.5fr 1fr` レイアウトに再構成し、担当者テーブルを 4 カラムに簡素化する

**Non-Goals**:

- 商談登録フォームのデザイン変更
- 顧客登録フォームのデザイン変更
- ビジネスロジックの変更
- 新規 Server Action の追加

## Decisions

### D1: MeetingInfoSection を 3 コンポーネントに分割する

**選択**: MeetingInfoSection から出席者とヒアリング情報を抽出し、以下の 3 コンポーネントに分割する:
- `MeetingInfoSection` — 基本情報（種別・日時・場所）の表示/編集。表示モード（読み取り）と編集モード（フォーム）を `useState` で切り替える
- `MeetingAttendeesSection` — 出席者管理（社内/外部をサブセクションで分離）。独立した保存機能を持つ
- `MeetingHearingSection` — ヒアリング情報（hearing タイプのみ表示）。100px ラベル + 1fr 値の grid レイアウト。独立した保存機能を持つ

**却下**: 1 コンポーネント維持 — デザインで出席者は右カラム、ヒアリングは左カラムに配置されるため、1 コンポーネントでは左右カラムへの分散配置ができない

**Rationale**: デザインの配置要件を満たすには物理的にコンポーネントを分離する必要がある。各コンポーネントが独立した保存機能を持つことで、ユーザーはセクション単位で変更を保存できる。

### D2: 分割後の各コンポーネントは updateMeetingAction を共有する

**選択**: 3 コンポーネントすべてが既存の `updateMeetingAction` を呼び出す。各コンポーネントは自身が管理するフィールドのみを FormData にセットする。
**却下**: コンポーネントごとに専用 Server Action を新設する

**Rationale**: `updateMeetingAction` は FormData に含まれるフィールドのみを更新する設計のため、部分更新が可能。Action の新設はスコープ外（ビジネスロジック変更に該当）であり、既存 Action の再利用で要件を満たせる。

### D3: 顧客一覧の不要クエリを削除する

**選択**: 顧客一覧から `contactCountMap`（担当者数）と `inquiryCountMap`（引き合い数）の取得ロジックを削除する。`dealCountMap`（案件数）は「関連案件数」カラムとして残るため維持する。
**却下**: クエリを残して非表示にする — 不要な DB アクセスがパフォーマンスに影響する

**Rationale**: デザインで削除された情報の取得は無駄なリソース消費となる。`clientRepository.countContactsByClientIds` の呼び出しと `inquiryRepository.findAllByOrganization` の呼び出しを削除し、`dealRepository.findAllByOrganization` のみ残す。

### D4: 顧客詳細の担当者テーブルは CSS grid で 4 カラム化する

**選択**: 担当者テーブルのカラムを 4 つに統合する（名前, 部署・役職, 連絡先, アクション）。`grid-template-columns: 1.2fr 1fr 1.4fr 120px` を使用する。部署と役職を「部署・役職」、メールと電話を「連絡先」として 1 カラムに統合する。
**却下**: 既存の table 要素を維持して列を非表示にする — カラム統合ができない

**Rationale**: 単純な列の非表示ではなくデータの統合（部署+役職、メール+電話）が必要なため、CSS grid ベースの実装に変更する。

### D5: 商談詳細の基本情報はヘッダーバー内で表示/編集を切り替える

**選択**: 既存のツールバー（`bg-bg-toolbar` の div）を拡張し、基本情報（種別・日時・場所）を読み取り表示する。「編集」ボタンで MeetingInfoSection をフォームモードに切り替え、「キャンセル」で表示モードに戻す。
**却下**: 常に編集可能なフォームとして表示する — デザインの読み取り表示要件と乖離する

**Rationale**: デザインではヘッダー部分に基本情報が読み取り表示され、編集は意図的なアクションとして分離されている。`useState<"display" | "edit">` で切り替えることで、フォームコンポーネントの再利用と読み取り表示の両立が可能。

## Risks / Trade-offs

**[Risk]** MeetingInfoSection の分割により、既存の保存フローが変わる（全フィールド一括保存 → セクション個別保存）
→ **Mitigation**: `updateMeetingAction` が部分更新に対応していることを実装時に確認する。対応していない場合は Action 側で未送信フィールドを更新対象から除外するよう修正する（既存挙動への影響は最小限）。

**[Risk]** 顧客一覧のクエリ削除により、将来カラムを復活させる場合にクエリの再実装が必要になる
→ **Mitigation**: 削除するクエリは単純な集計（`countContactsByClientIds`, `findAllByOrganization`）であり、必要時に容易に復元可能。YAGNI 原則に従い不要な時点で削除する。

**[Risk]** 担当者テーブルの table → CSS grid 変更により、編集モーダルの既存ロジックに影響する可能性がある
→ **Mitigation**: 編集モーダル（`editingContact` state）はテーブル表示とは独立しているため、表示部分の変更のみで完結する。追加フォームも同様に独立している。
